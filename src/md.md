## 1. ВВЕДЕНИЕ

### 1.1 Цель документа
Настоящий документ определяет требования к разработке системы умного контрольно-пропускного пункта (КПП) на базе технологий компьютерного зрения и машинного обучения.

### 1.2 Область применения
Система предназначена для автоматизации процессов контроля доступа на охраняемых объектах с использованием биометрической идентификации и распознавания транспортных средств.

### 1.3 Определения и сокращения
- **КПП** — контрольно-пропускной пункт
- **CV** — Computer Vision (компьютерное зрение)
- **API** — Application Programming Interface
- **ML** — Machine Learning (машинное обучение)
- **RTSP** — Real Time Streaming Protocol
- **Kafka** — распределенная платформа потоковой передачи данных

### 1.4 Целевая аудитория
- Разработчики системы
- Системные архитекторы
- Тестировщики
- Администраторы безопасности
- Заказчик

## 2. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

### 2.1 Контекст системы
Система умного КПП функционирует в режиме реального времени, обрабатывая видеопотоки с камер наблюдения для идентификации людей и транспортных средств. Система интегрируется с существующими системами контроля доступа и учёта рабочего времени.

### 2.2 Архитектурный подход
- **Архитектурный стиль**: Микросервисная архитектура
- **Развертывание**: On-premise
- **Модель обработки**: Асинхронная с использованием брокера сообщений Apache Kafka
- **API стиль**: RESTful API на базе FastAPI

### 2.3 Основные функциональные блоки
1. **Регистрация пользователей** — управление профилями и биометрическими данными
2. **Распознавание** — идентификация лиц и транспортных средств
3. **Контроль доступа** — принятие решений о предоставлении доступа
4. **Аудит и логирование** — фиксация всех событий системы

## 3. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 3.1 Модуль регистрации пользователей

#### 1.1 Регистрация нового пользователя
**Приоритет**: Высокий  
**Описание**: Система должна обеспечивать регистрацию новых пользователей с сохранением биометрических данных.

**Требования**:
- Ввод персональных данных (ФИО, должность, подразделение, контакты)
- Захват биометрических данных (минимум 5 фотографий лица в разных ракурсах)
- Создание эмбеддингов лиц с использованием нейросетевых моделей
- Сохранение данных в БД с шифрованием персональной информации
- Присвоение уникального идентификатора

#### 1.2 Управление профилями
**Приоритет**: Высокий  
**Требования**:
- Редактирование данных пользователя
- Обновление биометрических данных
- Деактивация/активация учетных записей
- Управление правами доступа (роли, зоны доступа, временные ограничения)

#### 1.3 Регистрация транспортных средств
**Приоритет**: Средний  
**Требования**:
- Привязка транспортных средств к пользователям
- Сохранение данных: номер, марка, модель, цвет
- Множественная привязка (один пользователь — несколько авто)

### 3.2 Модуль распознавания

#### 2.1 Распознавание лиц
**Приоритет**: Критический  
**Описание**: Система должна в режиме реального времени идентифицировать лица из видеопотока.

**Требования**:
- Обработка RTSP потоков от IP-камер (до 8 камер одновременно)
- Детекция лиц в кадре (алгоритмы: MTCNN, RetinaFace или аналоги)
- Извлечение признаков (face embeddings) с использованием моделей на базе ArcFace/FaceNet
- Сравнение с базой данных (поиск по сходству с порогом ≥95%)
- Время обработки одного кадра: ≤200ms
- Работа в различных условиях освещения
- Обработка частичных окклюзий (маски, очки)

#### 2.2 Распознавание номеров транспортных средств
**Приоритет**: Высокий  
**Требования**:
- Детекция и распознавание государственных регистрационных знаков
- Поддержка форматов: российские номера (ГОСТ Р 50577-2018)
- Распознавание при скорости до 60 км/ч
- Точность распознавания: ≥98%
- Работа в условиях: день/ночь, различная погода

#### 2.3 Обработка результатов распознавания
**Требования**:
- Сохранение изображений (лицо, кадр с авто) в хранилище
- Генерация событий распознавания с метаданными
- Публикация событий в Kafka для дальнейшей обработки

### 3.3 Модуль контроля доступа

#### 3.1 Принятие решений о доступе
**Приоритет**: Критический  
**Описание**: Система должна автоматически принимать решения о предоставлении доступа.

**Требования**:
- Проверка прав доступа пользователя (роль, зона, временной интервал)
- Проверка статуса учетной записи (активна/заблокирована)
- Проверка наличия в черном списке
- Формирование решения: РАЗРЕШИТЬ/ЗАПРЕТИТЬ/ТРЕБУЕТ_ПОДТВЕРЖДЕНИЯ
- Время принятия решения: ≤100ms

#### 3.2 Управление физическими барьерами
**Требования**:
- Интеграция с контроллерами шлагбаумов/турникетов
- Отправка команд открытия/закрытия
- Контроль состояния исполнительных устройств
- Обработка аварийных ситуаций (принудительное открытие при эвакуации)

#### 3.3 Уведомления и алерты
**Требования**:
- Уведомление охраны при запрете доступа
- Алерты при обнаружении лиц из черного списка
- Уведомления о технических сбоях
- Поддержка каналов: веб-интерфейс, email, Telegram

### 3.4 Модуль аудита и логирования

#### 4.1 Логирование событий
**Приоритет**: Высокий  
**Описание**: Система должна фиксировать все значимые события.

**Требования**:
- Логирование всех попыток доступа (успешных и неуспешных)
- Сохранение данных: timestamp, user_id, event_type, location, decision, confidence_score
- Сохранение изображений/видеофрагментов событий
- Структурированное логирование (JSON формат)

#### 4.2 Аналитика и отчетность
**Требования**:
- Генерация отчетов по шаблонам (посещаемость, нарушения, статистика)
- Экспорт данных (CSV, Excel, PDF)
- Визуализация данных (графики, дашборды)
- Поиск и фильтрация событий по критериям

#### 4.3 Хранение данных
**Требования**:
- Срок хранения логов событий: минимум 1 год
- Срок хранения изображений: минимум 90 дней
- Архивирование устаревших данных
- Обеспечение целостности данных (контрольные суммы)

## 4. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1 Производительность

- Поддержка до 10 000 зарегистрированных пользователей
- Поддержка до 50 000 записей в базе транспортных средств

#### 1.2 Время отклика
- Время распознавания лица: ≤200ms
- Время принятия решения о доступе: ≤100ms
- Время открытия API endpoint: ≤50ms (p95)
- Задержка в Kafka pipeline: ≤500ms (end-to-end)

### 4.2 Доступность и надежность

#### 2.1 Доступность системы

## 7. АРХИТЕКТУРА СИСТЕМЫ

### 7.1 Микросервисы

#### User Service
- **Назначение**: Управление пользователями и биометрией
- **Язык**: Python + FastAPI
- **База данных**: PostgreSQL
- **API endpoints**: /users, /biometrics, /vehicles

#### Recognition Service
- **Назначение**: Распознавание лиц и номеров
- **Язык**: Python + PyTorch/TensorFlow
- **Входные данные**: RTSP потоки, изображения
- **Выходные данные**: События в Kafka

#### Access Control Service
- **Назначение**: Принятие решений о доступе
- **Язык**: Python + FastAPI
- **База данных**: PostgreSQL + Redis (кэш)
- **Интеграция**: Контроллеры доступа (Wiegand/RS-485)

#### Audit Service
- **Назначение**: Логирование и аналитика
- **Язык**: Python + FastAPI
- **База данных**: PostgreSQL + Elasticsearch
- **API endpoints**: /events, /reports, /analytics

#### API Gateway
- **Назначение**: Единая точка входа, маршрутизация, аутентификация
- **Технология**: Kong / Traefik / nginx
- **Функции**: Rate limiting, JWT validation, load balancing

#### Notification Service
- **Назначение**: Отправка уведомлений и алертов
- **Язык**: Python
- **Каналы**: WebSocket, Email, Telegram API

### 7.2 Коммуникация между сервисами

#### Синхронная коммуникация
- REST API между сервисами (для запросов данных)
- HTTP/2 для оптимизации

#### Асинхронная коммуникация
- Kafka топики для событий:
  - `recognition.face` — события распознавания лиц
  - `recognition.vehicle` — события распознавания авто
  - `access.decision` — решения о доступе
  - `audit.events` — события аудита
  - `notifications` — уведомления

### 7.3 Потоки данных

#### Поток распознавания лица
1. Камера → RTSP поток → Recognition Service
2. Recognition Service → детекция и распознавание
3. Recognition Service → публикация события в Kafka (`recognition.face`)
4. Access Control Service → подписка на Kafka → принятие решения
5. Access Control Service → команда контроллеру доступа
6. Audit Service → подписка на Kafka → логирование

#### Поток регистрации пользователя
1. Web UI / API → API Gateway → User Service
2. User Service → сохранение данных в PostgreSQL
3. User Service → генерация эмбеддингов лиц
4. User Service → сохранение биометрии

## 8. МОДЕЛИ МАШИННОГО ОБУЧЕНИЯ

### 8.1 Распознавание лиц

#### Детекция лиц
- **Модель**: RetinaFace или MTCNN
- **Входные данные**: RGB изображение
- **Выходные данные**: Координаты bbox, landmarks (глаза, нос, рот)

#### Извлечение признаков (Face Embeddings)
- **Модель**: ArcFace (ResNet-100 backbone) или FaceNet
- **Входные данные**: Выровненное изображение лица (112×112)
- **Выходные данные**: Вектор эмбеддингов (512-dim)

#### Сравнение лиц
- **Метод**: Cosine similarity
- **Порог**: 0.6-0.7 (настраиваемый)
- **База данных**: Vector DB (Faiss, Milvus) для быстрого поиска

### 8.2 Распознавание номеров

#### Детекция номерных знаков
- **Модель**: YOLOv8 или Faster R-CNN
- **Входные данные**: RGB изображение
- **Выходные данные**: Координаты bbox номерного знака

#### OCR (распознавание текста)
- **Модель**: CRNN + CTC loss или Transformer-based OCR
- **Входные данные**: Изображение номерного знака
- **Выходные данные**: Текст номера + confidence score

### 8.3 Требования к моделям
- Точность распознавания лиц: ≥95% (FAR ≤1%)
- Точность распознавания номеров: ≥98%
- Скорость инференса: ≤200ms на изображение (с GPU)
- Поддержка batch обработки для оптимизации

## 15. ГЛОССАРИЙ

**Биометрические данные** — физиологические и биологические характеристики человека (в данном контексте — изображения лиц), используемые для идентификации.

**Эмбеддинг (Face Embedding)** — векторное представление лица в многомерном пространстве признаков, генерируемое нейросетью.

**RTSP (Real Time Streaming Protocol)** — протокол потоковой передачи видео в реальном времени.

**Kafka** — распределенная платформа для потоковой обработки данных, используется для асинхронной коммуникации между микросервисами.

**Микросервис** — архитектурный стиль, при котором приложение строится как набор небольших независимых сервисов.

**On-premise** — развертывание на собственной инфраструктуре заказчика (в отличие от облака).

**API Gateway** — единая точка входа для всех API запросов, обеспечивает маршрутизацию, аутентификацию, rate limiting.

**Wiegand** — стандартный протокол передачи данных между считывателями карт доступа и контроллерами.

**JWT (JSON Web Token)** — стандарт для создания токенов доступа, используется для аутентификации API.

**Cosine Similarity** — метрика сходства векторов, используется для сравнения face embeddings.
